# Code audit — July 2026

Full evaluation of canonia v0.1.0 (2026-07-08): three independent review passes
(security, correctness, architecture) over `src/` + `tests/` + CI. Every
correctness finding was **reproduced against a scratch canon** before being
listed — these are verified behaviors, not speculation. This file is the durable
record; keep statuses current as items are fixed.

**Overall verdict: architecture sound-with-caveats.** Clean module DAG, one
write-path validation chokepoint (`_save → _validate`), transport-agnostic
`CanonService`, honest degradation, derived state correctly disposable, safe
YAML everywhere, parameterized SQL, clean subprocess use, OIDC trusted
publishing. The caveats cluster in three places: **concurrency is oversold**
(last-writer-wins, no conflict detection), **provenance/attribution is the
weakest subsystem** (dead identity plumbing, autocommit off by default,
timestamp data loss), and **derived-state divergence has no detection**
(index/site never updated on MCP writes).

## Blockers

| # | Status | Finding |
|---|--------|---------|
| B1 | **fixed** 2026-07-08 | **Stored XSS in the site index page.** `site.py` inlines `json.dumps` of raw titles/summaries into `<script>const INDEX = …`; `json.dumps` doesn't escape `</script>`, so agent-authored content closes the script block and injects markup/JS into `index.html`. The body renderer (`markdown_html.py`) escapes correctly — this path bypasses it. Fix: escape `<` as `\u003c` in the inlined JSON (plus U+2028/U+2029). |
| B2 | **fixed** 2026-07-08 | **Duplicate ids across domains, with shadowing.** `create` checks `_path_for(concept).exists()` (per-domain path), not graph membership. `create(id="foo", domain="infra")` succeeds when `process/foo.md` exists; `Graph.load` iterates `sorted(rglob)` so whichever file sorts first wins all reads — an impostor can silently replace the canon's answer for an existing id. Fix: check `id in graph.concepts`. |
| B3 | **fixed** 2026-07-08 | **`created:`/`updated:` frontmatter silently destroyed on every rewrite.** Both keys are in `KNOWN_KEYS` (schema.py) so they're excluded from `extra`, but no `Concept` field holds them and `frontmatter()` never emits them. Every server write and importer re-emit drops them. Fix: model the fields, round-trip the raw YAML value. |
| B4 | **fixed** 2026-07-08 | **`import --commit` writes (and `--prune` deletes) even when the gates fail.** `cli.py cmd_import` computes issues, then writes unconditionally; the verdict only sets the exit code. Also the gate validates the emitted set **in isolation**, so a body ref to a concept already in the canon false-fails, while real breakage gets written. Fix: gate the write (override via `--force`); validate the predicted post-commit canon (emitted ∪ existing for non-prune; emitted alone for prune). |
| B5 | **fixed** 2026-07-08 | **Non-atomic writes + `remove` gate ignores inline refs.** (a) `path.write_text` with no temp+rename — a crash mid-write leaves a truncated concept file (server `_save`, importer `plan.write`). (b) `graph.dependents()` checks `references`/`redirect`/`superseded_by` but not inline `[[id]]` body refs, while the validate gate *does* — so `remove` can leave a canon failing its own gate. |
| B6 | **fixed** 2026-07-08 | **No path containment beyond the id regex + unpinned model download.** (a) `_path_for` joins `concepts_dir / domain / id.md` with no `is_relative_to` check; the kebab regex is the only traversal defense and `schema.id_pattern` is user-configurable — one loosened config reinstates arbitrary file write/delete via MCP tools. (b) The ONNX model + vocab are fetched over HTTPS from a hardcoded HF repo but with **no checksum verification**; a repointed upstream file gets cached and executed by onnxruntime. Pinned SHA-256s (verified against the upstream git-LFS pointers): model `afdb6f1a0e45b715d0bb9b11772f032c399babd23bfc31fed1c170afc848bdb1`, vocab `07eced375cec144d27c900241f3e339478dec958f92fddbc551f295c992038a3`. |

## Other correctness findings (verified, unfixed)

**Medium:**
- **Zero-config slug collisions silently lose content** — `setup.md` and
  `sub/setup.md` both slugify to `setup`; `plan.write()` writes the same target
  twice, second wins. Detect id collisions during planning. *(Partially
  mitigated once B4 blocks gate-failing commits — the duplicate-id gate fires —
  but dedup/error at plan time is the real fix.)*
- **`merge` is non-transactional** — `_save(tgt)` runs before `_save(src)`; if
  the source fails validation (e.g. legacy `source: []`), the target has
  already absorbed provenance on disk with no tombstone written. Validate both
  before writing either.
- **Semantic index keeps a stale `domain` column** after a domain-only move
  (selection is purely content-hash). Domain-filtered semantic search then
  wrongly includes/excludes the concept. Also: `meta` stores model name/dim but
  nothing validates them on open — switching `index.model` without a rebuild
  silently mixes embeddings.
- **Hybrid search down-ranks freshly-written concepts**: once any index exists,
  search blends 50/50; an un-indexed new concept gets sim=0 so its max score is
  0.5 vs 1.0 — the newest knowledge systematically loses until the next
  `canonia index build`. No staleness detection/warning exists anywhere.
- **`markdown_html` double-escapes `&`** in link hrefs and wikilink aliases
  (`[AT&T](…?a=1&b=2)` → broken URL, `AT&amp;amp;T` labels).
- **Dangling-ref gate flags `[[id]]` inside fenced code blocks**
  (`extract_inline_refs` has no fence awareness, unlike `iter_headings`).

**Low:** merging into an archived concept makes the source vanish from search
entirely (tombstone + archived both excluded); merging an id that's already a
tombstone silently re-targets it; `update(references="abc")` becomes
`['a','b','c']` — three *valid* one-char ids (no type enforcement against the
tool inputSchema); punctuation-only queries return arbitrary concepts at score
0; `limit=0` silently becomes 1; `init --domains "process, lore"` creates a
`concepts/ lore` dir (leading space) mismatching the parsed domain; default id
pattern accepts `foo-`/`foo--bar` and `$` allows a trailing newline, and a
user-supplied pattern isn't required to be end-anchored; CRLF `vocab.txt` on
Windows tokenizes everything to `[UNK]` (`rstrip("\n")` only); `--source
name=C:\repo` misparses on Windows drive colons; stale site pages are never
deleted from `site/c/` on rebuild; dotfile filter misses `.md` files inside
hidden *directories*; `initialize` echoes back any client `protocolVersion`
instead of negotiating.

## Security posture (beyond B1/B6)

- **Threat model:** `canonia serve` is stdio-only — no network listener exists
  anywhere in the tree. The attacker-controlled surface is agent-authored
  concept content flowing into the write path, the static site, and git.
- **Low-severity open items:** curated importer resolves `../`-escaping
  `path:` values from mapping.yml (arbitrary file read into concept bodies —
  normalize + contain in `sources.py`); absolute host paths leak in tool
  results/error messages (return canon-relative paths); no message-size cap on
  the stdio loop (memory DoS); GitHub Actions pinned to mutable tags and the
  PyPI publish action to a **branch** — pin to commit SHAs; markdown links
  accept protocol-relative `//evil.com` hrefs.
- **Done right (keep it that way):** `yaml.safe_load`/`safe_dump` at all four
  call sites; every sqlite query parameterized; `subprocess.run(["git", …])`
  with `add --` separator, no `shell=True`, never pushes; body renderer
  escapes-first and neutralizes `javascript:` links; model URL hardcoded (config
  can't redirect it), no archive extraction; PyPI Trusted Publishing via OIDC
  with tag↔version gate; `init` writes a `.gitignore` for `.canonia/` + `site/`
  and `build` prints the no-auth warning.

## Architecture caveats (accepted-for-now list)

- **Concurrency:** "optimistic concurrency" is currently last-writer-wins.
  `update` is whole-file read-modify-write with no version check; `create` has
  a TOCTOU between exists-check and write; concurrent autocommits can collide
  on git's index.lock. Cheapest real fix: content-hash compare-and-swap
  (return a hash from `get`, accept it in `update`, reject on mismatch).
- **Identity/attribution is dead code:** `Identity` is constructed nowhere;
  the server always runs as `ANONYMOUS`; the `--author` git path is
  unreachable; stdio offers no auth channel. With autocommit **off by default**,
  MCP writes are anonymous uncommitted working-tree changes — "git provides
  versioning + authorship (human AND agent)" is aspirational until an identity
  channel (env var / CLI flag / future HTTP transport) + autocommit-on land.
- **Derived-state divergence:** the semantic index and site are only updated
  by manual CLI runs; no staleness detection. The content-hash infrastructure
  for index invalidation already exists in `index.py`.
- **Seams, honestly:** `access.py` is a real seam but **read-side only** — no
  `can_write` call sites, so write-RBAC/review-queue needs new seams. The
  sqlite-vec "backend seam" is capability *reporting* only (no dispatch point;
  `EmbeddingIndex` is instantiated directly). `site.generator` and `mcp_name`
  config keys are parsed and never read (dead).
- **Id namespacing is mildly anti-designed:** flat kebab ids are baked into
  filenames, site URLs, sqlite PKs, and the wikilink regexes in `markdown.py` +
  `markdown_html.py`. Namespacing later is a migration, not a config change —
  **reserve a separator character (reject `.`/`:` inside ids) now.**
- **Read path scale wall:** every tool call re-reads and re-YAML-parses every
  concept file (`Graph.load` per request; `create` pays it twice). Fine at
  ~150, seconds-per-call at 10k. An mtime-based cache in `Graph.load` (already
  the single chokepoint) fixes it with no architectural change. Brute-force
  cosine is NOT the bottleneck — ignore it below ~10k vectors.
- **Error-handling nits:** `except TypeError` around tool dispatch mislabels
  TypeErrors raised *inside* handlers as "invalid arguments"; semantic-search
  failures degrade silently with zero stderr signal (one log line would fix
  operability).

## Test-coverage gaps

Covered since the audit (tests/test_cli.py + additions elsewhere): CLI import
gate paths, `created`/`updated` round-trip, cross-domain duplicate-id create,
inline-ref remove gate, XSS neutralization, atomic-write temp cleanup, path
containment under a loosened id pattern. Still untested: concurrency
(everything); post-write hybrid-ranking behavior; server error paths
(restore-on-tombstone, broken-redirect get, TypeError mislabel); config edge
cases (null `domains:`, null `hybrid_weight:` → TypeError); the rest of the
CLI (init/validate/index/build, `_parse_sources` Windows drive-colon parse).

## Roadmap suggestions (from the audit)

1. **Trust layer before any adoption push:** identity channel → revives
   `--author`; autocommit default **on** (it's the only audit trail); LLM-kind
   identities write as `status: draft` by default (`Identity.kind` exists for
   exactly this); hash CAS on update.
2. **Strategic, cheap now / expensive later:** reserve id namespace separator;
   wire index invalidation off the existing content hashes (or stop
   down-weighting un-indexed concepts); mtime cache in `Graph.load`.
3. **Positioning honesty** (from the strategy review): the write side
   (canonical, id-addressable, lifecycle-managed units) is the defensible
   claim; the read side is RAG and must be measured, not assumed. Agent
   write-back without a review state is a liability to foreground, not a
   headline feature.
